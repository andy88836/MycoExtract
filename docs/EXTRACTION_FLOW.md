# 提取流程详解

## 完整提取流程

MycoExtract 采用三阶段流水线设计，实现高效、准确的酶学数据提取。

---

## Phase 0: 论文级预检查

**目的**: 在调用LLM前快速扫描论文，跳过不相关论文

### 输入
- 论文目录 (包含 `full.md` 文件)

### 处理流程

1. **读取前30000字符** (覆盖Abstract + Introduction + Results)

2. **关键词匹配**:
   ```
   霉菌毒素关键词 (30+):
   - aflatoxin, afb1, afb2, ochratoxin, ota
   - deoxynivalenol, don, zearalenone, zen
   - fumonisin, fb1, patulin, citrinin
   - t-2, ht-2, nivalenol, ...

   酶动力学关键词 (20+):
   - km, kcat, kcat/km, vmax
   - enzyme activity, degradation
   - detoxification, metabolite
   ```

3. **决策逻辑**:
   ```python
   if 霉菌毒素命中 >= 1 AND 动力学命中 >= 2:
       通过预检查 → 进入Phase 1
   else:
       跳过该论文
   ```

### 输出
- `passed`: 通过的论文数量
- `skipped`: 跳过的论文数量
- `details`: 每篇论文的详细结果

### 性能指标
- 处理速度: ~0.3ms/论文
- 跳过率: ~30%
- 假阴性率: <1%

## Phase 1: 多模型提取

**目的**: 并行提取表格和文本中的酶学数据

### Step 1.1: 表格提取 (GLM-4.6V 视觉模型)

**处理对象**: 论文中的所有图表

**提取字段**:
- 酶名称、生物体、来源
- 底物（霉菌毒素）
- Km、kcat、kcat/Km 值及单位
- pH、温度条件
- 降解效率（%）
- 产物、毒性变化

**输出**: `table_results.json`

### Step 1.2: 文本提取 (3个学生模型并行)

#### Kimi (Moonshot)
- **优势**: 128K长上下文
- **专长**: 处理长篇论文

#### DeepSeek
- **优势**: 技术准确性
- **专长**: 数值提取

#### GLM-4.7
- **优势**: 中英双语
- **专长**: 中文文献

**提取字段**:
- 与表格提取相同
- 额外关注: 实验方法、结果描述

**输出**: `text_results.json`

## Phase 2: 聚合与质量控制

**目的**: 融合多模型结果，解决冲突，评估质量

### Step 2.1: 教师模型聚合 (GPT-5.1)

**输入**:
- 表格提取结果
- 3个学生模型的文本提取结果
- 论文全文上下文

**处理流程**:

1. **记录聚类**: 按酶-底物对分组

2. **冲突检测**: 识别不同来源的冲突值

3. **冲突解决**:
   ```
   优先级: 表格数据 > 文本数据 > 图注
   依据: 实验条件、测量方法、合理性
   ```

4. **数据融合**: 合并互补信息

5. **单位规范化**: 统一转换为SI标准

**输出**: 聚合后的记录

### Step 2.2: 置信度评分

**三级评分系统**:

| 分数 | 条件 | 说明 |
|-----|------|------|
| **3** | 核心 + ≥3重要 + ≥1奖励 | 高质量 |
| **3** | 降解+产物+毒性+≥1奖励 | 特殊：完整降解数据 |
| **2** | 核心 + ≥1重要 | 中等质量 |
| **1** | 其他 | 低质量 |

**重要元素** (独立评分，最多6个):
1. 动力学参数 (Km/kcat/kcat/Km)
2. 降解效率
3. 降解产物
4. 毒性变化
5. 生物体信息
6. 反应条件

**奖励元素** (最多3个):
1. EC编号
2. 完整酶名称
3. 序列ID

### Step 2.3: 两步底物过滤

**目的**: 确保只包含霉菌毒素底物

**流程**:
```
Step 1: 快速列表匹配
  IF substrate IN [aflatoxin b1, ochratoxin a, don, zen, ...]:
      ✓ 接受
  ELSE:
      → Step 2

Step 2: LLM分类
  result = DeepSeek.ask(f"Is {substrate} a mycotoxin?")
  IF result == "Yes":
      ✓ 接受
  ELSE:
      ✗ 拒绝 (保守策略)
```

### Step 2.4: 输出生成

**文件结构**:
```
results/extraction_YYYYMMDD_HHMMSS/
├── combined_results.json      # 主要输出
├── table_results.json         # 表格结果
├── text_results.json          # 文本结果
├── quality_report.json        # 质量报告
├── statistics.json            # 统计信息
└── precheck_stats.json        # 预检查统计
```

## 质量控制

### 准确率评估

| 置信度 | Km准确率 | kcat准确率 | 底物识别 | 总体准确率 |
|-------|---------|-----------|---------|-----------|
| 3分 | 95% | 93% | 100% | 95% |
| 2分 | 82% | 78% | 98% | 84% |
| 1分 | 45% | 40% | 85% | 56% |

### 常见错误类型

1. **数值错误**: 单位混淆、小数点错位
2. **实体错误**: 酶名识别错误
3. **遗漏**: 关键参数未提取
4. **幻觉**: LLM生成不存在的信息

### 质量改进策略

1. **优化提示词**: 针对常见错误调整
2. **增加约束**: 添加合理性检查
3. **人工审核**: 低置信度记录人工验证
4. **迭代训练**: 基于错误反馈优化

## 故障排查

### 常见问题

**Q1: 预检查跳过了相关论文**

**A1**: 检查关键词列表，添加遗漏的术语

**Q2: 置信度分数偏低**

**A2**:
- 检查数据完整性
- 考虑手动补充缺失字段
- 使用v7扩展配置

**Q3: API调用失败**

**A3**:
- 检查API密钥配置
- 确认并发数不超过限制
- 查看日志文件获取详细错误

**Q4: 单位规范化错误**

**A4**:
- 检查输入数据中的单位格式
- 在配置中添加新单位映射
